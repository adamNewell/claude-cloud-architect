# arch-docs

Mine documentation for architectural intent that code alone cannot reveal. Extracts FLOW, DEBT, and RISK CANDIDATE tags from ADRs, design documents, READMEs, and Confluence exports.

**CRITICAL — rlmgrep does not exist:** Any reference to `rlmgrep` in older stubs or documentation is incorrect. This skill uses `qmd` exclusively for all documentation search. Do not attempt to install or invoke `rlmgrep`.

**Core principle:** Documentation captures the *why* behind architectural decisions. Code shows what was built; docs show what was decided, what risks were accepted, and what debt was knowingly incurred. These are distinct data sources — arch-docs mines the intent layer that static analysis cannot reach.

## NEVER

- **NEVER index `node_modules/`, `dist/`, `build/`, `.next/`** — generated output produces thousands of noise hits that swamp architectural signal. Always filter with `--mask`.
- **NEVER raise confidence above 0.60 because a document merely names a component** — only raise confidence when the document explicitly states a decision, risk, or debt acknowledgment (explicit heading, consequences section, or "we decided to..."). Name-dropping is not architectural intent.
- **NEVER treat README marketing copy as architectural intent** — "Getting Started", "Installation", "Features" sections describe the product, not the architecture. Tagging these inflates CANDIDATE counts with false positives.
- **NEVER skip `qmd collection remove` after a session** — orphaned collections persist globally across all projects in `~/.cache/qmd/index.sqlite`. Accumulation degrades search relevance for unrelated projects over time.
- **NEVER treat a stale ADR as current architectural fact** — ADRs older than 2 years or explicitly superseded by another ADR describe historical decisions, not the current architecture. Note the age in `source_evidence`; do not raise confidence.
- **NEVER promote a doc-sourced CANDIDATE tag without confirming the described component exists in code** — documentation describes intended architecture; code is the ground truth. The promotion step is the verification gate.
- **NEVER use `rlmgrep`** — it does not exist. Any stub or reference to it is incorrect. `qmd` is the sole documentation mining tool.

## Before Invoking arch-docs

**Ask yourself:**
- Does the target repository have documentation worth mining? Check for `docs/`, `doc/`, `adr/`, or `ARCHITECTURE.md` before indexing.
- Are you looking for architectural *intent* or *implementation*? (Intent = arch-docs; Implementation = arch-structure/arch-search)
- Has a previous arch-docs run already indexed this repo in the current session? (If yes, skip `collection add` — query the existing collection.)
- Are there ADRs or design docs that explain why major structural choices were made? (These produce the highest-value findings.)

## What to Mine (Priority Order)

Mine these document types in order — earlier types have higher architectural signal:

1. **ADRs** (`docs/adr/*.md`, `docs/decisions/*.md`, `**/adr-*.md`, `**/*-adr.md`) — explicit decision records with context and consequences
2. **Design docs** (`DESIGN*.md`, `design*.md`, `docs/design/`, `docs/*.md`) — flow descriptions, trade-off analysis, planned migrations
3. **READMEs and architecture docs** (`README.md`, `ARCHITECTURE.md`, `docs/architecture/`) — high-level overview, known limitations, technical debt callouts
4. **Confluence exports** (`confluence/*.html`, `*.html` in doc dirs) — note: `qmd` indexes `.md` by default; HTML requires explicit `--mask "**/*.html"` or pre-conversion to Markdown

**What to skip:**
- `node_modules/`, `dist/`, `build/`, `.next/` — generated output, not architectural intent
- Autogenerated API docs (e.g., Swagger/OpenAPI HTML, typedoc output) — schema, not decisions
- Changelogs and release notes — operational history, not design rationale
- Marketing or onboarding copy in READMEs (e.g., "Getting Started", "Installation") — boilerplate, not architectural intent

## qmd Workflow (Index → Search → Cleanup)

The complete qmd workflow for arch-docs has three phases: index the docs directory, query for architectural signals, then clean up the ephemeral collection.

### Phase 1 — Index

```bash
# Identify the docs root (use the most specific directory that contains docs)
DOCS_DIR="$REPO/docs"          # typical location; adjust per repo
COLLECTION="arch-docs-$SESSION"  # session-scoped name prevents cross-session collisions

# Add the collection (indexes .md files by default)
qmd collection add "$DOCS_DIR" --name "$COLLECTION" --mask "**/*.md"

# For repos with ADRs at repo root alongside docs/
# Index the whole repo but mask to doc files only:
qmd collection add "$REPO" --name "$COLLECTION" --mask "{docs,doc,adr,decisions}/**/*.md"

# Optional: add semantic context to improve reranking relevance
qmd context add "$DOCS_DIR" "Architecture decisions, design rationale, technical debt, risk assessments"

# Generate vector embeddings for semantic search (required for qmd query and qmd vsearch)
qmd embed
```

**If docs directory is empty or does not exist:** Fall back to the Read tool. Scan for `.md` files manually:

```bash
# Check for doc files before indexing
find "$REPO" -name "*.md" -not -path "*/node_modules/*" -not -path "*/dist/*" | head -20
```

If fewer than 3 `.md` files exist outside `node_modules`, skip qmd indexing entirely and use the Read tool to open files directly. Note in the session that doc coverage is absent.

### Phase 2 — Search

Run all three query families. Each targets a different tag kind.

**FLOW tags — operations, sequences, data movement:**

```bash
# Recommended: query mode (BM25 + expansion + semantic reranking)
qmd query "data flow pipeline processing sequence" -c "$COLLECTION" --json -n 10

# Keyword-only fallback (faster, no LLM; use when embeddings not yet built)
qmd search "flow diagram event sequence processing" -c "$COLLECTION" --json -n 10
```

**DEBT tags — technical debt, known limitations, deferred work:**

```bash
qmd query "technical debt migration deprecated legacy TODO FIXME workaround" -c "$COLLECTION" --json -n 10
qmd search "tech debt known limitation future work trade-off deferred" -c "$COLLECTION" --json -n 10
```

**RISK tags — security decisions, vulnerabilities, accepted risks:**

```bash
qmd query "security risk vulnerability accepted risk trade-off threat" -c "$COLLECTION" --json -n 10
qmd search "security decision known vulnerability risk mitigation" -c "$COLLECTION" --json -n 10
```

**Retrieve full documents for high-scoring hits (score >= 0.75):**

```bash
# Get full text of a specific document by qmd:// path from JSON results
qmd get "qmd://$COLLECTION/decisions/adr-001.md"

# Get first 60 lines (for long ADRs)
qmd get "qmd://$COLLECTION/decisions/adr-001.md" -l 60

# Bulk-retrieve all ADRs at once
qmd multi-get "qmd://$COLLECTION/adr/*.md" -l 50 --json
```

**Score guidance:**
- `>= 0.80` — high-confidence architectural signal; read full doc and write CANDIDATE tag
- `0.60–0.79` — read the snippet; confirm the doc section is decision-level content before writing tag
- `< 0.60` — skip unless the snippet excerpt is clearly architectural intent (e.g., explicit "Decision:" section)

### Phase 3 — Cleanup

Remove the ephemeral collection after tagging to avoid stale index accumulation:

```bash
qmd collection remove "$COLLECTION"
```

Do not skip cleanup. Persistent collections accumulate across sessions and degrade search relevance for unrelated projects.

## Search Mode Selection

| Mode | Command | When to Use |
|------|---------|-------------|
| `qmd query` | `qmd query "<text>" -c $COLLECTION --json` | Primary — best relevance; requires embeddings (`qmd embed`) |
| `qmd search` | `qmd search "<text>" -c $COLLECTION --json` | Fallback when `qmd embed` has not run; keyword-only BM25 |
| `qmd vsearch` | `qmd vsearch "<text>" -c $COLLECTION --json` | Semantic-only without reranking; use when query is slow |

**Always prefer `qmd query`** unless embeddings have not been built. `qmd search` is reliable for exact-match queries (e.g., "ADR-007", "migration", "deprecated"). `qmd vsearch` is useful for concept-level queries when keyword overlap is low.

**For advanced flags, output formats, and multi-get patterns not shown here — MANDATORY: READ ENTIRE FILE `../../cookbook/qmd/cli.md` before using unfamiliar flags.**
**Do NOT load** `../../cookbook/tag-store/queries.md` unless you need to query existing tags after writing — it covers SQL templates, not qmd commands.

## Writing CANDIDATE Tags

Write one tag per distinct architectural finding. Do not batch unrelated findings into a single tag.

Build the tag JSON as a variable first to avoid shell quoting errors:

```bash
# Build tag JSON separately — avoids fragile mixed-quote patterns
TAG_JSON=$(cat <<EOF
{
  "kind": "FLOW",
  "subkind": "<specific-flow-type>",
  "target_repo": "$REPO",
  "target_ref": "<doc-path-relative-to-repo>",
  "name": "<descriptive-name>",
  "description": "<what-the-doc-says-about-this-flow>",
  "confidence": 0.60,
  "weight_class": "MACHINE",
  "status": "CANDIDATE",
  "source_evidence": "<verbatim-snippet-from-doc>",
  "session_id": "$SESSION"
}
EOF
)

bun tools/tag-store.ts write --tag "$TAG_JSON" --session "$SESSION"
```

### Tag Kind Rules

| Kind | Subkind examples | Source signals |
|------|-----------------|----------------|
| `FLOW` | `DATA_PIPELINE`, `EVENT_SEQUENCE`, `REQUEST_LIFECYCLE`, `BATCH_PROCESS` | ADRs describing operations; design docs with sequence diagrams or flow descriptions; README sections explaining how data moves |
| `DEBT` | `DEFERRED_MIGRATION`, `KNOWN_LIMITATION`, `LEGACY_COUPLING`, `TODO_ARCHITECTURAL` | ADR consequences sections with deferred decisions; README "Known Issues" or "Limitations"; design docs explicitly naming trade-offs accepted |
| `RISK` | `SECURITY_DECISION`, `KNOWN_VULNERABILITY`, `COMPLIANCE_GAP`, `SCALABILITY_CONCERN` | Security ADRs; design docs with risk sections; README warnings about authentication, authorization, or data exposure |

**Confidence rule:** All arch-docs tags use `confidence=0.60` as baseline. Raise to `0.70` for explicit "Decision:", "Risk:", or "Debt:" headings. Raise to `0.80` only if the document verbatim names an architectural component referenced in the codebase.

**weight_class:** Always `MACHINE`. Documentation mining is probabilistic — the agent infers architectural significance from text; it is not deterministic analysis.

**status:** Always `CANDIDATE`. Promote to `VALIDATED` only after confirming the described system component exists in code.

## Distinguishing Architectural Intent from Boilerplate

Not all documentation is architectural. Apply this filter before writing a tag:

**Write a tag if the document section:**
- Names an explicit decision and its rationale ("We chose X over Y because...")
- Describes a data flow or process sequence that is not obvious from code
- Acknowledges technical debt, a workaround, or a deferred refactoring
- Identifies a known risk, security trade-off, or compliance gap
- Explains a constraint imposed by a third-party system or organizational requirement

**Do NOT write a tag if the section:**
- Is a "Getting Started" or installation guide
- Lists dependencies without explaining why they were chosen
- Describes API endpoints or configuration options without design rationale
- Is a generated table of contents, badge list, or CI status section
- Repeats information already captured as a VALIDATED tag by arch-structure

## Pre-flight Check

```bash
# Verify qmd is installed
qmd --version  # expected: qmd 1.0.7 or later

# Verify docs directory exists and has content
ls "$REPO/docs" 2>/dev/null || echo "No docs/ directory — check for README.md and ARCHITECTURE.md at repo root"

# Count candidate doc files before indexing
find "$REPO" -name "*.md" \
  -not -path "*/node_modules/*" \
  -not -path "*/dist/*" \
  -not -path "*/.archimedes/*" | wc -l
```

If `qmd` is not found: it is not installed. Do not fall back to `rlmgrep` — it does not exist. Use the Read tool to open documents directly.

## PDF Limitation

`qmd` indexes `.md` and `.html` files only. PDF files (e.g., architecture decision PDFs, exported Confluence pages) cannot be indexed by `qmd`. For PDF documents:
- Use the Read tool to open the PDF directly if it is within the repository
- Note the limitation in the session: "PDF docs present but not indexed — manual review required"
- Do not attempt to convert PDFs with external tools unless the user explicitly requests it

## Handling Missing or Sparse Documentation

| Situation | Action |
|-----------|--------|
| `docs/` directory does not exist | Check repo root for `README.md`, `ARCHITECTURE.md`; read them with the Read tool |
| Fewer than 3 `.md` files found | Skip qmd indexing; read files directly with the Read tool |
| Only autogenerated docs found | Note "no human-authored architectural docs found"; skip tagging |
| Only marketing copy in README | Note "README contains no architectural intent"; skip tagging |
| Confluence exports as HTML | Re-index with `--mask "**/*.html"` or convert to Markdown first |

## Complete Example

```bash
REPO="/path/to/service"
SESSION="arch-$(date +%s)"
COLLECTION="arch-docs-$SESSION"

# Phase 1: Index
qmd collection add "$REPO/docs" --name "$COLLECTION" --mask "**/*.md"
qmd context add "$REPO/docs" "Architecture decisions, design docs, ADRs, technical debt"
qmd embed

# Phase 2: Query
qmd query "technical debt migration deprecated" -c "$COLLECTION" --json -n 10 > /tmp/debt-results.json
qmd query "data flow sequence pipeline" -c "$COLLECTION" --json -n 10 > /tmp/flow-results.json
qmd query "security risk vulnerability accepted" -c "$COLLECTION" --json -n 10 > /tmp/risk-results.json

# Review high-scoring hits (score >= 0.75)
cat /tmp/debt-results.json | jq '.[] | select(.score >= 0.75) | {score, file, snippet}'

# Get full ADR for a high-scoring hit
qmd get "qmd://$COLLECTION/adr/adr-005-caching-strategy.md"

# Write a DEBT tag from the finding (use heredoc to avoid quoting errors)
TAG_JSON=$(cat <<EOF
{
  "kind": "DEBT",
  "subkind": "DEFERRED_MIGRATION",
  "target_repo": "$REPO",
  "target_ref": "docs/adr/adr-005-caching-strategy.md",
  "name": "Redis migration deferred",
  "description": "ADR-005 documents deferred migration from in-memory cache to Redis; accepted as tech debt until traffic scales beyond 10k RPS",
  "confidence": 0.70,
  "weight_class": "MACHINE",
  "status": "CANDIDATE",
  "source_evidence": "Decision deferred: Redis integration planned for Q3 but accepted risk of in-memory cache loss on pod restart",
  "session_id": "$SESSION"
}
EOF
)
bun tools/tag-store.ts write --tag "$TAG_JSON" --session "$SESSION"

# Phase 3: Cleanup
qmd collection remove "$COLLECTION"
```

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| `collection not found` | Collection name mismatch or removed | Re-run `qmd collection add`; verify with `qmd collection list` |
| 0 results from `qmd query` | Embeddings not built | Run `qmd embed` before querying |
| `qmd vsearch` returns nothing | Embeddings absent | Run `qmd embed -f` to force re-embedding |
| Low scores across all queries (< 0.50) | Docs don't contain architectural intent | Apply the boilerplate filter; use Read tool for manual review |
| `qmd embed` slow | Large doc set | Normal; 900-token chunks with 15% overlap take time on CPU |
| HTML files not indexed | Default mask is `**/*.md` | Re-add with `--mask "**/*.html"` or convert to Markdown |
| PDF files not indexed | qmd does not support PDF | Use Read tool directly; note limitation in session |
| `rlmgrep: command not found` | rlmgrep does not exist | Do not install — use `qmd` exclusively |
